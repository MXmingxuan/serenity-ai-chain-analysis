# AI 供应链瓶颈研究 Agent 框架

本项目当前包含 `000.md` 到 `009.md` 十个提示词。它们不是十个彼此独立的问题，而是一条用于研究 AI 产业链瓶颈机会的 prompt chain。

核心目标：

> 寻找 AI 超级周期中“需求加速 + 供给刚性 + 利润可兑现 + 市场未充分定价”的产业链瓶颈环节。

本框架的作用是把这套提示词变成可重复运行的 agentic research workflow：每一轮都有明确输入、搜索任务、证据要求、输出报告和下一轮状态。

## 1. 总体原则

1. 严格按 `000 -> 009` 顺序运行。
2. 每一步只解决一个研究问题。
3. 每一步必须读取上一轮报告和累计状态。
4. 搜索结果必须进入证据账本，而不是只写进正文。
5. 每一轮都输出独立报告。
6. 不得把“相关公司”等同于“受益公司”。
7. 不得把“市场叙事”等同于“产业事实”。
8. 找不到强证据时，结论必须降级。
9. 最后一轮才做综合判断。
10. 最终产物必须能沉淀为可复盘研究卡片。

## 2. 推荐目录结构

每次研究一个对象时，在 `research_runs/` 下新建一个运行目录：

```text
research_runs/
  YYYY-MM-DD-研究对象/
    state/
      research_state.yaml
      evidence_ledger.csv
    reports/
      000_scope.md
      001_value_chain.md
      002_demand.md
      003_supply.md
      004_targets.md
      005_evidence_audit.md
      006_pricing.md
      007_disconfirmation.md
      008_final_report.md
    cards/
      009_research_card.md
    sources/
      source_index.md
```

当前项目根目录的 `000.md` 到 `009.md` 作为原始提示词，不建议改写。每次运行时把它们视为固定研究协议。

## 3. Prompt Chain 地图

|步骤|文件|研究问题|核心输出|搜索强度|
|---|---|---|---|---|
|0|`000.md`|研究对象是什么，应该按什么方式展开|对象定义、初始假设、研究路线|低|
|1|`001.md`|它在 AI 产业链第几阶传导|产业链传导图、上下游、传导强度|中|
|2|`002.md`|需求是否真实加速|需求来源、需求证据、持续性判断|高|
|3|`003.md`|供给能否快速跟上|瓶颈识别、供给弹性评分、利润转化判断|高|
|4|`004.md`|哪些公司真正受益|候选公司、A/B/C 分类、利润兑现路径|高|
|5|`005.md`|哪些结论有证据，哪些只是叙事|证据等级、事实/推理/叙事拆分|中|
|6|`006.md`|市场是否已经定价|股价、估值、拥挤度、回撤风险|高|
|7|`007.md`|什么信号说明我们错了|产业/技术/财务/订单/股价反证|中高|

|9|`009.md`|如何沉淀成个人研究资产|研究卡片|低|

## 4. 主控 Agent 职责

主控 agent 不直接跳到投资结论。它负责调度研究链条。

每一轮执行顺序：

1. 读取当前 prompt。
2. 读取上一轮报告。
3. 读取 `state/research_state.yaml`。
4. 读取 `state/evidence_ledger.csv`。
5. 生成本轮搜索计划。
6. 搜索公开资料。
7. 记录来源和证据等级。
8. 输出本轮报告。
9. 更新研究状态。
10. 写出下一轮必须验证的问题。

主控 agent 必须维护三类状态：

- `claims`: 当前已经形成的核心判断。
- `open_questions`: 后续需要验证的问题。
- `evidence`: 支持或反驳判断的证据。

## 5. 搜索优先级

不同阶段使用不同来源。搜索时优先使用原始、权威、可追溯信息。

### 5.1 需求验证优先来源

用于 `002.md`。

1. hyperscaler capex 指引。
2. 云厂商、GPU/ASIC、数据中心、下游客户财报。
3. earnings call transcript。
4. 公司公告、投资者交流材料。
5. 行业机构数据。
6. 订单、长约、预付款、客户锁产能、交期变化。
7. 多个产业链环节的交叉验证。

### 5.2 供给瓶颈优先来源

用于 `003.md`。

1. 公司产能、扩产计划、capex。
2. 技术路线、良率、认证周期、设备交付周期。
3. 主要供应商和新增竞争者。
4. 价格、交期、产能利用率。
5. 替代材料、替代技术、替代供应链。

### 5.3 公司映射优先来源

用于 `004.md`。

1. 年报、10-K、20-F、招股书。
2. 分部收入、客户结构、毛利率。
3. backlog、订单、长约、预付款。
4. 投资者交流、电话会、公告。
5. 同业公司交叉验证。

### 5.4 市场定价优先来源

用于 `006.md`。

1. 股价区间表现。
2. 当前估值和历史估值区间。
3. 同业估值比较。
4. 成交额、换手率。
5. 卖方覆盖和社媒热度。
6. 财报后股价反应。

## 6. 证据等级规则

所有关键证据必须进入 `evidence_ledger.csv`。

|等级|含义|例子|
|---|---|---|
|S|直接产业证据|长约、预付款、客户入股、产能售罄、涨价、交期翻倍|
|A|公司经营证据|收入加速、毛利率改善、订单 backlog 增长、capex 扩张|
|B|行业交叉验证|多家公司电话会、权威机构数据、多个环节一致反馈|
|C|弱证据|单篇研报、社媒观点、零散传闻、主题映射|
|D|证据不足|只有概念，没有数据或事实支持|

强制规则：

- 没有 S/A/B 证据，不得称为“确认瓶颈”。
- 只有 C/D 证据，只能写为“待验证叙事”。
- 如果证据只能证明需求增长，不能自动推出利润兑现。
- 如果公司收入占比不清，标的优先级必须降级。
- 如果股价和估值已经明显透支，最终动作不得直接写“高优先级研究”，除非反证充分说明仍有预期差。

## 7. 每轮输出契约

每一轮报告必须包含：

1. 本轮任务边界。
2. 使用的上下文。
3. 搜索计划。
4. 关键证据表。
5. 分析正文。
6. 本轮结论。
7. 下一步需要继续验证的问题。
8. 本轮新增或更新的证据等级。

如果本轮不适合搜索，例如 `008.md` 和 `009.md`，也要说明“不引入新信息，仅整合前序报告”。

## 8. 质量门槛

进入下一轮前，主控 agent 需要自检：

- 当前结论有没有明确证据来源？
- 是否混淆事实、推理和市场叙事？
- 是否保留了反证和负面证据？
- 是否把“主题相关”误写成“利润受益”？
- 是否说明了证据缺口？
- 是否输出了下一步问题？

如果以上任一项缺失，当前轮次需要补充后才能进入下一步。

## 9. 推荐执行方式

推荐采用半自动方式，而不是一开始就写完整 runner。

第一阶段：手工 agentic run

- 给定一个研究对象。
- 按 `000 -> 009` 逐轮运行。
- 每轮保存报告、证据账本和状态。
- 人工检查关键证据是否可信。

第二阶段：模板稳定化

- 从 1-2 个案例中总结常用搜索 query。
- 固化证据账本字段。
- 固化报告模板。
- 固化最终研究卡片格式。

第三阶段：工具化 runner

- 只有当手工链条稳定后，再考虑自动 runner。
- runner 的重点不是自动生成结论，而是自动创建目录、读取 prompt、传递状态、检查输出结构。

## 10. 使用口令

当需要开始一个研究对象时，可以直接使用：

```text
请按 AGENT_FRAMEWORK.md 执行一轮新的 AI 供应链瓶颈研究。
研究对象：【填写对象】
要求：按 000-009 逐轮运行，每轮保存报告，维护 evidence_ledger 和 research_state，最后输出综合报告和研究卡片。
```

如果只想先跑前半段，可以使用：

```text
请按 AGENT_FRAMEWORK.md 先运行 000-004，只做到标的初筛，不做最终投资结论。
研究对象：【填写对象】
```

如果只想做证据审计，可以使用：

```text
请基于已有 reports，按 005.md 做证据分级，重点拆分事实、推理、叙事和未验证假设。
```

## 11. 结构化市场数据层

`006.md` 涉及股价涨幅、估值状态、成交额、换手率和拥挤度。这里不能主要依赖搜索摘要。搜索适合判断卖方覆盖、社媒共识和市场叙事；价格、成交、换手、估值等数字应优先来自结构化数据源。

### 11.1 数据源分工

|市场|优先数据源|备选数据源|主要用途|
|---|---|---|---|
|A 股|Tushare|交易所公告、公司公告、搜索核验|日线、复权价格、成交额、换手率、估值、财务指标|
|港股|Yahoo Finance CLI / Tushare 港股接口|交易所披露、公司公告|价格、成交额、市值、换手率、部分估值|
|美股|Yahoo Finance CLI / Tushare 美股接口|SEC、公司 IR、搜索核验|价格、成交额、市值、估值、财务数据|
|指数 / 行业|Tushare 指数和行业接口|Yahoo Finance、公开指数资料|行业涨跌幅、板块强度、相对收益|

### 11.2 Tushare 使用规则

本项目已安装 `tushare` skill 和 Python 包。使用结构化数据前需要先做环境检查：

1. Python 可用。
2. `tushare` 包可导入。
3. `TUSHARE_TOKEN` 存在于环境变量或本机 Tushare 配置。
4. 如果接口权限、积分或覆盖范围不足，必须在报告中说明限制。
5. 不得因为缺数据而让模型编造估值、成交额、换手率或涨跌幅。

Token 安全规则：

- 不把 token 写入 Markdown、CSV、YAML 或代码仓库。
- 优先从 `TUSHARE_TOKEN` 环境变量读取。
- 如果必须临时配置，只在本机私密环境中配置，不进入研究报告。

### 11.3 Yahoo Finance CLI 使用规则

本项目已安装两个 Yahoo Finance skill 说明目录：

- `skills/yahoo-finance`
- `skills/yahoo-finance-cli`

同时，本项目提供本地可运行工具：

```powershell
uv run --script tools/yf AAPL
uv run --script tools/yf history NVDA 1y
uv run --script tools/yf fundamentals MSFT
uv run --script tools/yf compare AAPL,MSFT,GOOGL
uv run --script tools/yf search "taiwan semiconductor"
```

使用边界：

- Yahoo Finance CLI 不需要 API key。
- 适合美股、港股、ETF、加密货币、外汇和部分海外市场。
- Yahoo Finance 可能限流；如果失败，应等待后重试，并在 `market_data_notes.md` 记录。
- Yahoo 的估值和财务字段可能存在缺失或口径差异，需要保留 `source=Yahoo Finance via yfinance`。
- 海外股票如港股需使用 Yahoo 代码格式，例如 `0700.HK`。

### 11.4 第 006 步的新执行顺序

运行 `006.md` 前，先执行结构化市场数据准备：

1. 读取 `004_targets.md` 中的 A 类和 B 类标的。
2. 写入或更新 `symbol_universe.csv`。
3. 标准化股票代码和市场。
4. 按市场选择数据源。
5. 拉取日线、成交额、换手率、市值和估值指标。
6. 计算近 1 / 3 / 6 / 12 个月涨跌幅。
7. 计算成交额和换手率的历史分位或异常放大信号。
8. 输出 `market_data/price_snapshot.csv`。
9. 输出 `market_data/valuation_snapshot.csv`。
10. 输出 `market_data/liquidity_snapshot.csv`。
11. 再用搜索补充卖方覆盖、社媒讨论和热门共识。
12. 最后写 `006_pricing.md`。

### 11.6 结构化数据输出字段

`price_snapshot.csv` 建议字段：

```csv

```

`valuation_snapshot.csv` 建议字段：

```csv

```

`liquidity_snapshot.csv` 建议字段：

```csv

```

### 11.7 拥挤度判断规则

拥挤度不只看股价涨幅。`006_pricing.md` 需要综合：

- 近 1 / 3 / 6 / 12 个月涨幅。
- 股价相对 52 周高低点。
- 成交额是否明显放大。
- 换手率是否明显提高。
- 估值是否处于历史高位。
- 卖方覆盖是否增加。
- 社媒讨论是否升温。
- 财报或公告后是否出现利好不涨、小利空大跌。

建议分级：

|等级|含义|
|---|---|
|低拥挤|价格、成交和讨论热度都未明显升温|
|中等拥挤|价格已有表现，但估值和讨论尚未极端|
|高拥挤|价格、成交、估值、讨论至少三项明显升温|
|极度拥挤|估值透支、交易过热、叙事高度一致，且利好反应变弱|

### 11.8 数据和搜索的边界

结构化数据回答“是多少”：

- 涨跌幅是多少。
- 成交额是多少。
- 换手率是多少。
- 估值是多少。
- 历史分位是多少。

搜索和 agent 判断回答“为什么”：

- 为什么市场开始关注。
- 卖方是否形成共识。
- 社媒是否过热。
- 当前涨幅是否来自业绩兑现还是叙事扩散。
- 利好是否已经被价格提前反映。

第 006 步最终报告必须同时引用结构化数据和非结构化搜索结果。两者冲突时，应保留冲突并说明可能原因，不得强行合并成单一结论。
